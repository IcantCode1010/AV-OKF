# Changelog

All notable changes to this project are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).

## Unreleased

### Changed

- Recorded the Stage 6.5 OKF retrieval architecture correction: OKF-routed chat should crawl/read the exported `knowledge/` bundle directly (`index.md`, frontmatter, Markdown links, relations, `source_manifest.md`, and `log.md`) instead of requiring approved OKF topics to be embedded into the RAG vector database. The existing `okf_topic` RAG projection is now treated as legacy/optional cache; raw RAG remains the raw document discovery layer, while OKF remains the reviewed Markdown/YAML source of truth for agents.
- Completed the Stage 6 router fallback path: low-confidence rules-first chat routes can now call the workspace LLM provider to classify the question into `okf_only`, `rag_only`, `hybrid`, `missing_context`, or `unsupported`; high-confidence safety routes remain rules-first. The fallback is fail-closed: missing keys, provider failures, malformed JSON, or unsupported route/category values keep the original conservative rule decision. Chat traces now preserve `routerMode` so reviewers can tell whether a reply was routed by rules or by the LLM fallback.
- Made chat OKF-first end to end, matching the platform's core three-layer intent (approved OKF governs, RAG fills gaps, and both stay swappable behind stable contracts for the eventual agent). `routeChatQuestion` now accepts either a bare question string or the structured `{question, workspaceId, conversationContext}` input from `docs/architecture/query-router.md`, and `production-chat-service.ts` threads the last few session turns through as that context. Router rules were widened to route plain interrogative questions (`what is`, `how does`, `explain`, `describe`) to OKF at medium confidence — previously only exact keywords like "definition" or "official" matched, so natural questions (e.g. "what is DC generation") always fell through to the generic "I need a little more context" reply regardless of content. Hybrid retrieval now reads OKF before RAG sequentially instead of firing both in parallel, and an `okf_only` route with zero approved evidence downgrades to labeled RAG discovery (explicitly framed as unreviewed, both in the deterministic reply and the LLM prompt) instead of dead-ending. The trace gained `approvedOkfAvailable`, `ragUsedForDiscoveryOnly`, and `finalEvidenceStatus` fields, and citations now carry `coveredByOkfConceptIds` — the OKF-priority signals the Stage 7 validator will read. Fixed the LLM answer builder's citation-marker instruction along the way: the prompt described the `[n]` requirement in prose without an example, and `gpt-4o-mini` reliably omitted markers on short discovery answers, always tripping the citation-validation fallback; the prompt now states the rule as a hard requirement with a worked example.
- Added the Stage 6c LLM answer builder (`chat-answer.ts`): when the workspace has an LLM provider key saved in Settings (Anthropic or OpenAI, reusing the topic-enrichment key), chat replies are synthesized from the retrieved evidence instead of echoing excerpts. Guardrails: the model receives only the numbered evidence excerpts and must return `{answer, supported}` JSON; answers with no `[n]` citation markers or markers outside the retrieved set are rejected; rejection, provider failure, malformed responses, and keyless workspaces all fall back to the deterministic excerpt echo, and error/missing-evidence replies never call the LLM. A `supported: false` verdict returns an explicit "related but doesn't directly answer" reply rather than forcing an answer. The agent trace now records `answerMode` (`llm`/`deterministic`) with provider/model, shown as a badge in the Sources & Trace panel. Retrieval results now carry fuller non-persisted `evidence` excerpts (1500 chars) for prompting, separate from the short persisted citation excerpts (240 chars). Live-verified end-to-end with a real OpenAI key: a RAG-routed question produced a synthesized, correctly-cited reply traced as `LLM answer · gpt-4o-mini`.
- Chat message threads now auto-scroll the newest message into view on load and after each reply (`chat-thread.tsx`); previously the scroll container started at the top and the latest reply was hidden below the fold.

- Added the Stage 6b chat retrieval slice (`chat-retrieval.ts`): `sendMessage` now runs real retrieval instead of a canned per-route string. `okf_only` queries fetch only approved `okf_topic` RAG chunks, `rag_only` queries fetch `raw_extraction` chunks regardless of review status, and `hybrid` queries run both concurrently and concatenate the results (OKF first). The reply is built deterministically from the retrieved excerpts with `[n]` citation markers matching the existing citation-chip UI, and falls back to an explicit missing-evidence message per route when nothing is found instead of guessing. The router trace's `retrievalToolsCalled`/`sourcesRead` fields are now populated instead of hardcoded empty. Source-type/approval filtering happens client-side in `chat-retrieval.ts` since `RetrievalRequest.filters.sourceTypes`/`reviewStatus` are declared on the type but not yet applied by `rag-repository.ts`. LLM-synthesized prose answers remain a follow-up slice; this pass only wires retrieval, citations, and trace. Live-verified against a local Postgres/Redis stack, which surfaced a real crash: `retrieveDocuments` throws when the embedding provider is unavailable (missing/invalid `OPENAI_API_KEY`, budget exceeded, transient failure), which was taking down the whole chat page with an unhandled Server Action error. `runChatRetrieval` now catches retrieval failures and returns a distinct `retrievalError` result so the reply says retrieval is temporarily unavailable instead of crashing or claiming there's no evidence. Re-verified end-to-end with a real OpenAI API key against a full local stack (Postgres/Redis/MinIO, extraction worker, `next dev`): uploaded a fixture PDF, let it extract and index for real, and confirmed `rag_only` chat returns a genuine hybrid-search citation grounded in the actual extracted text.
- Reframed the upcoming topic workflow: after PDF upload and text extraction, an LLM should review the extracted document text, propose draft topic records with source sections/pages, let the reviewer tweak those topics, optionally enhance an individual topic with the LLM, and only then allow human approval/export to OKF.
- Added workspace-level Anthropic API key settings for future AI enrichment. Keys are encrypted at rest, managed from Settings, and never displayed back to the client after save.
- Added on-demand LLM topic enrichment for unapproved topics, including side-by-side raw/enriched review, explicit raw-vs-enriched approval choice, and a full audit trail of prompt, raw response, provider, model, requester, success state, and error message.
- Added OpenAI as a second LLM enrichment provider alongside Anthropic. Workspace Settings now offers a provider dropdown instead of a fixed Anthropic field; `topic-enrichment.ts` selects the matching provider implementation via a factory keyed on the workspace's saved provider, and switching providers replaces the single stored key rather than keeping both.
- Condensed the document detail page into a tree-nav layout: `document-tree-nav.tsx` presents Summary/Metadata/Extraction/Topics/Logs as a collapsible tree (with a flattened mobile view), and the previous ~950-line monolithic page now renders panel content from the new `document-detail-panels.tsx`.
- Added approved-OKF-topic-to-RAG sync (`okf-rag-sync.ts`): approved topics are indexed as their own `okf_topic` search chunks, separate from raw extraction chunks, with content-hash change detection to skip re-syncing unchanged topics. Triggered from the admin Reindex page, which now also reports synced/unchanged/failed counts. Search results display an "OKF topic" vs "raw extraction" source badge.
- Implemented Stage 5 OKF-to-RAG coverage links (`okf-coverage.ts`): on export against the production backend, approved topics resolve which active raw-extraction RAG chunks overlap their source pages, write that list to `covered_rag_chunk_ids`/`coverage_type` frontmatter, and sync it into `OkfConceptChunkLink` (stale links removed on re-export). `searchKeyword`/`searchVector` now populate `RetrievalResult.coveredByOkfConceptIds` from that table instead of hardcoding `[]`. Local JSON-vault exports skip resolution, since they have no RAG chunks.
- Added extraction and topic-generation activity feedback to the document detail page: a header line shows extraction start/finish timestamps and page count on every tab (not just Extraction), the Summary tab's extraction tile shows real counts/timestamps instead of a static string, and the document now defaults to the Topics tab (with an actionable "Generate topics" prompt) as soon as extraction completes, instead of staying on an unchanging Summary tab. "Generate topics" now shows a completion banner reporting the resulting topic count.
- Improved Stage 3 heading-detection quality after a real 737 QRH document (402 pages) produced 368 topics, most single-page with obscure titles like `"Lights.Index.5"` or bare `"0.1"`. `topic-records.ts` now: requires numbered headings to have real trailing title text (not just a bare digit); rejects bare page-index/cross-reference codes (`"0.1"`), single-token dotted codes (`"Lights.Index.5"`), dot-leader index/TOC entries (`"LOW QUANTITY.......13.13"`), and bare 1-3 letter lines (alphabetical-index dividers like `"D"`, or truncated print artifacts like `"REV"` that precede the real heading on the next line); and assigns `medium` confidence (previously unused) to the weaker short-line heading heuristic, reserving `high` for ALL-CAPS/explicitly-numbered matches. Verified against the real QRH document: 368 → 341 → 332 topics, with the originally-reported junk patterns eliminated.
- Added the Stage 6 chat UI foundation: real, persisted `ChatSession`/`ChatMessage` records (new Prisma models, `production-chat-repository.ts`/`production-chat-service.ts`/`chat-backend.ts` following the existing `document-backend.ts` facade pattern), a live "Chat" nav entry, and an Immersive-layout conversation view (`/chat`, `/chat/[sessionId]`) with a full-width message thread, inline numbered citation chips, and a standing "Sources & Trace" side panel that collapses into the existing `Sheet` component on narrow viewports. Sending a message stores it and returns a stubbed placeholder assistant reply with empty citations and a null trace — the query router, OKF/RAG retrieval, and real trace data are deliberately out of scope for this pass. Chat requires the production Postgres backend; local JSON-vault dev mode shows an explicit unavailable notice rather than a fake local store.

### Fixed

- Chat sends now refresh the active conversation immediately after the Server Action persists the user/assistant messages. `sendChatMessageAction` revalidates and redirects back to the current `/chat/[sessionId]`, fixing the browser-tested case where the response was saved to Postgres but did not appear until a manual reload.
- Cleaned up the Turbopack NFT trace warning path by setting the app tracing root explicitly, suppressing the known `next.config.ts` trace issue, and splitting OKF relation-only types/vocabulary reads away from the runtime relation validator. This keeps document detail relation dropdowns, OKF export, bundle retrieval, and relation validation on the same behavior while avoiding unnecessary dynamic file reads in Server Component bundles.
- Hardened Stage 6.5 OKF bundle matching so unrelated sparse-bundle questions no longer surface an approved OKF card just because they share common words like "system" or "check" with an approved topic. `retrieveOkfBundleEvidence` now filters stopwords/generic retrieval terms, requires meaningful title/description/source-metadata matches before an OKF result can qualify, blocks body-only matches from becoming approved evidence, and records `matchedTerms`/`matchStrength`/`matchReason` for future agent trace and validation work. Added regressions proving unrelated queries such as "galley water heater leak troubleshooting" and "cabin pressurization system checks" do not match `Main Gear Brake System`, and that low-relevance OKF misses fall back to raw RAG instead of rendering `APPROVED - OKF`.
- Chat session pages now follow the standard chat layout pattern: fixed-height conversation surface, scrollable message thread, centered empty state, and a bottom-anchored composer instead of letting the input float high when a conversation has few messages.
- Document library rows were intermittently unclickable. Root cause was Next.js 16's client-side router silently failing to commit some navigations (reproduced even on a bare `next/link` click with no custom code involved). `document-library.tsx` now navigates with a plain `<a href>` and `window.location.assign()` instead of `next/link` and `router.push()`, forcing a full browser navigation for this list.
- `document-row-navigation.ts`'s click-ignore selector didn't exclude anchor tags, so a click landing on the title link could double-fire both the link's own navigation and the row's programmatic navigation to the same URL. Anchor tags are now excluded.
- Removed an `approved` OKF topic (`32-apu-ecu-monitors-apu-performance-parameters-to-determine-*.md`) whose title/description was raw, truncated extraction text rather than a real section heading. The source pages have a multi-column layout that the Stage 2 heading detector misread; the topic is now `rejected` in the vault fixture and removed from the knowledge bundle's `index.md`/`source_manifest.md`, with the retraction recorded in `log.md`.
- Docker Compose's `web` and `worker` services didn't load `apps/web/.env`, so search embeddings had no provider credentials at runtime; both services now load that env file, and the compose file no longer separately threads a stale `OPENAI_API_KEY` default.
